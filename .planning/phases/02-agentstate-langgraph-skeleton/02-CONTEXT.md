## Phase Boundary

Define the shared `AgentState` TypedDict and build a full LangGraph `StateGraph` with 10 placeholder async node functions, conditional routing edges (including the correction loop), and in-memory session isolation via `MemorySaver`. No real LLM calls â€” all nodes log and return empty dicts. The skeleton must compile and be traversable end-to-end in tests.

</domain>

<decisions>
## Implementation Decisions

### Node File Organization
- Node functions live in `agents/nodes/` subdirectory (matches Phase 1's nested pattern: `database/connectors/`)
- One file per node: `agents/nodes/gatekeeper.py`, `schema_linker.py`, `query_planner.py`, `sql_generator.py`, `executor.py`, `correction_plan.py`, `correction_sql.py`, `formatter.py`, `evaluator.py`
- Each file exports a single `async def <name>_node(state: AgentState) -> dict` function
- `agents/nodes/__init__.py` re-exports all node functions for clean graph builder imports: `from agents.nodes import gatekeeper_node, schema_linker_node, ...`
- Placeholder bodies: log which node was called, return `{}` (no state changes). Graph traverses fully without LLM.

### AgentState Field Design
- Use `Annotated[list, add_messages]` for the `messages` field â€” LangGraph's built-in accumulator, correct for multi-turn conversations
- Intermediate results are `Optional` with `None` defaults: `query_plan`, `generated_sql`, `db_results`, `error_log`, `final_answer`
- Required fields with defaults: `messages` (Annotated), `user_query: str = ""`
- Two additional fields beyond PROJECT.md schema:
  - `db_type: str = "sqlite"` â€” dialect for SQL generation, passed in at invocation
  - `query_type: Optional[str] = None` â€” set by gatekeeper to `"sql"` or `"conversational"`, drives first conditional edge
  - `db_manager: Optional[object] = None` â€” DatabaseManager instance passed in at invocation

**Final AgentState schema:**
```python
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    user_query: str
    db_type: str
    db_manager: Optional[object]
    query_type: Optional[str]
    schema: Optional[dict]
    relevant_tables: Optional[list]
    query_plan: Optional[str]
    generated_sql: Optional[str]
    db_results: Optional[list]
    error_log: Optional[str]
    retry_count: int
    final_answer: Optional[str]
```

### Correction Loop Logic
- **Max retries: 2** (3 total attempts: original + 2 corrections) â€” matches PROJECT.md spec
- **Trigger condition:** `error_log is not None` after executor node â†’ route to `correction_plan`
- **Exhaustion:** When `retry_count >= 2` and still failing â†’ route to `formatter` with error in `final_answer`
- **State reset on retry:** Clear only `generated_sql` and `db_results`. Preserve `error_log` and `query_plan` (correction_plan node reads error_log for diagnosis)
- **Conditional edge from executor:**
  - `error_log is not None AND retry_count < 2` â†’ `correction_plan`
  - `error_log is not None AND retry_count >= 2` â†’ `formatter` (exhausted)
  - `error_log is None` â†’ `formatter` (success)

### Memory & Session Isolation
- **Checkpointer:** `MemorySaver` (in-memory) â€” zero setup, sessions scoped to process lifetime. Phase 9 upgrades to SQLite persistence.
- **Thread ID format:** UUID4 string (`str(uuid.uuid4())`). Generated by caller before invoking graph.
- **Graph instance:** Module-level singleton in `graph/builder.py` â€” `compiled_graph = build_graph()` at import time. One graph per process.
- **Database injection:** `db_manager` field in `AgentState`. Caller sets it at invocation: `compiled_graph.invoke({"db_manager": dm, "user_query": q, "db_type": "sqlite"}, config={"configurable": {"thread_id": thread_id}})`

### Claude's Discretion
- Exact logging format inside placeholder nodes (use Python `logging` module, standard format)
- `graph/conditions.py` routing function names and signatures
- Test structure within `tests/graph/` (pytest fixtures, conftest patterns)
- Whether to use `END` directly or an explicit terminal node after formatter/evaluator

## Specific Ideas

- The 10 node sequence: `gatekeeper â†’ schema_linker â†’ query_planner â†’ sql_generator â†’ executor â†’ (correction_plan â†’ correction_sql â†’ executor loop) â†’ formatter â†’ evaluator â†’ END`
- Gatekeeper is the entry point â€” it sets `query_type`. If `query_type == "conversational"`, graph skips to formatter directly (short-circuit edge).
- Evaluator is the final node before END â€” it will eventually compute ragas scores but is a passthrough in Phase 2.
